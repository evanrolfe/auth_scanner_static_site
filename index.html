Features

How does it work?

Auth Scanner crawls your site while logged in under different users and roles. It attempts to access restricted paths while logged in as unprivileged users and generates a report based on the findings. The report lets you verify that restricted paths are properly locked down and only accessible to the right users.

 
 

Why is access control important?

Broken Access Control is listed as the fifth most prevalent web application vulnerability in the OWASP Top 10. It can lead to exposure of user's data and allow unprivileged users to gain admin access to parts of your application. Causes of this kind of vulnerability typically lie in your application logic so it cannot be prevented in the same way that vulnerabilities like SQL injection can be prevented by using a database abstraction layer.


Why use Auth Scanner?

  • Minimal configuration required: simply give it a set of user credentials and configure the login process (i.e. via an HTML form) then let it run.
  • Non-intrusive: Only GET requests are crawled, no forms are submitted.
  • Cross platform: Auth Scanner tests your web app as a blackbox so does not care what development platform you are using.
  • In-depth reporting: View detailed request & response logs for each path tested.
  • Monitoring of authorization: Continously test your access control to monitor for changes in behaviour and potential misconfigurations.

 

How does it integrate into my CICD pipeline?

Scans are triggered via our API, once a scan is completed, any changes in behaviour from the previous scan will be flagged and sent to your web-hook callback so they can be processed by your CICD pipeline. A typical process will look like:

  1. A pull-request has been opened
  2. The CICD Pipeline deploys code change to a staging server
  3. A scan on the staging server is triggered
  4. Any changes in authorization behaviour are sent back to the CICD pipeline
  5. If nothing is flagged by the scan then the code can be deployed to staging
